### 3. 今日关键问题与解答（一问一答）

#### Q1：为什么要用 Lua 替换原来的 Redis 预扣库存？

A：原来的 Redis 预扣库存是多步操作：

```text
hasKey
set
decrement
rollback
```

高并发首次访问时可能出现多个请求都判断 key 不存在，然后重复初始化库存，导致库存被回拨。

Lua 可以把这些操作放进 Redis 内部一次执行：

```text
初始化库存
判断是否重复
判断库存是否足够
扣库存
记录用户已抢
```

Redis 执行 Lua 脚本时不会被其他命令插队，所以能减少并发窗口。

---

#### Q2：Lua 中 `KEYS` 和 `ARGV` 是什么？

A：

```text
KEYS 用来接收 Redis key
ARGV 用来接收普通参数
```

例如 Java 传入：

```java
Arrays.asList(stockKey, userKey),
memberUsername,
quantity,
initialStock,
ttlSeconds
```

Lua 中对应：

```lua
KEYS[1] = stockKey
KEYS[2] = userKey
ARGV[1] = memberUsername
ARGV[2] = quantity
ARGV[3] = initialStock
ARGV[4] = ttlSeconds
```

注意 Lua 数组下标从 `1` 开始，不是 Java 的 `0`。

---

#### Q3：Lua 里的 `tonumber` 是干什么的？

A：Redis 传给 Lua 的参数通常是字符串。

例如：

```text
ARGV[2] = "1"
```

如果要做数字比较：

```lua
quantity <= 0
stock < quantity
```

就要先用：

```lua
tonumber(ARGV[2])
```

把字符串转成数字。

---

#### Q4：Lua 脚本里的 `redis.call` 是什么？

A：`redis.call` 用来在 Lua 脚本中执行 Redis 命令。

例如：

```lua
redis.call('GET', stockKey)
```

等价于 Redis 命令：

```redis
GET sms:seckill:stock:1
```

常见命令：

```text
EXISTS：判断 key 是否存在
SET：设置值
EXPIRE：设置过期时间
SISMEMBER：判断集合中是否有某个成员
DECRBY：扣减库存
SADD：加入集合
INCRBY：恢复库存
SREM：从集合移除成员
```

---

#### Q6：取消订单后为什么再次秒杀仍然提示重复？

A：因为 Lua 成功秒杀后会执行：

```lua
SADD sms:seckill:users:{activityId} memberUsername
```

取消订单时，原补偿逻辑只做了：

```text
Redis 库存 + quantity
```

没有执行：

```text
SREM sms:seckill:users:{activityId} memberUsername
```

所以用户仍然在 Redis users 集合里。

再次秒杀时 Lua 执行：

```lua
SISMEMBER userKey memberUsername
```

返回 1，于是判断为重复秒杀。

修复方式：

```text
取消/超时补偿时同时执行 INCRBY + SREM
```

---

#### Q7：之前做的补偿业务到底是干什么的？

A：补偿业务是为了处理：

```text
秒杀成功但交易未完成
```

例如：

```text
用户取消订单
订单超时未支付
订单创建过程中失败
```

补偿要恢复被占用的秒杀资源：

```text
秒杀记录 ORDER_CREATED -> CLOSED
Redis 库存 + quantity
Redis users 集合 SREM 用户
活动 sold_count 回退
```

否则库存会被无效订单一直占用。

---

#### Q8：补偿业务为什么要做幂等？

A：因为订单关闭可能被多个入口触发：

```text
用户手动取消
定时任务超时关闭
重复请求
异常重试
```

如果没有幂等保护，库存可能被恢复多次。

当前补偿通过条件更新实现幂等：

```text
只有 status = ORDER_CREATED 的记录才能改成 CLOSED
```

如果更新影响行数为 0，说明已经补偿过，直接 return。

#### Q11：为什么还保留数据库插入异常兜底？

A：因为 Redis 是高性能入口拦截，但数据库仍然是持久化事实来源。

可能出现：

```text
Redis key 过期
Redis 被清空
Redis 和数据库状态不一致
```

所以即使 Redis Lua 已经防重复，数据库记录和插入异常兜底仍然有意义。

可以理解为：

```text
Redis：高性能第一道门
MySQL：持久化兜底
```

#### Q12：MQ 在秒杀里解决什么问题？

A：MQ 主要解决：

```text
削峰
异步
解耦
失败重试
```

同步下单时，大量秒杀成功请求会同时创建订单，数据库压力大。

异步下单时：

```text
秒杀入口只做 Redis Lua 预扣和发送 MQ
消费者按数据库可承受速度创建订单
```

这样可以保护数据库。

#### Q13：异步秒杀为什么接口返回 PROCESSING？

A：因为引入 MQ 后，订单不是接口线程立即创建的，而是消费者异步创建。

所以接口不能马上返回订单 ID，只能返回：

```text
秒杀请求已受理，处理中
```

最终结果通过：

```text
WebSocket 推送
HTTP 查询秒杀结果
```

来获取。

A：

RabbitMQ：

```text
轻量
成熟
Spring Boot 集成简单
路由模型灵活

```

```text

```

当前项目建议用 RabbitMQ，因为：

```text

```

### 5. 面试高频点（含标准答案）

#### 1. 你为什么用 Lua 优化 Redis 秒杀库存？

答：

> 基础版中 Redis 预扣库存是多次操作，比如判断 key 是否存在、初始化库存、扣减库存，这些操作之间存在并发窗口。优化后我使用 Lua 脚本，把库存初始化、重复秒杀判断、库存判断、扣减库存、记录用户已抢合并成一个 Redis 原子操作。Redis 执行 Lua 脚本时不会被其他命令打断，可以减少并发下库存不一致和重复秒杀的问题。

---

#### 2. Lua 脚本怎么判断用户是否重复秒杀？

答：

> 我为每个秒杀活动维护一个 Redis Set，key 是 `sms:seckill:users:{activityId}`。用户秒杀成功时，Lua 脚本会执行 `SADD` 把用户名加入集合。下次请求时，通过 `SISMEMBER` 判断用户是否已经在集合中，如果存在就返回重复秒杀错误。这样重复校验可以在 Redis 中完成，减少数据库压力。

---

#### 3. 取消订单后为什么要 `SREM` 用户集合？

答：

> 因为当前业务允许用户取消订单后重新参与秒杀。Lua 成功后会把用户加入已抢集合，如果取消订单时只恢复库存，不把用户从集合中移除，那么再次秒杀仍然会被 Redis 判断为重复。所以补偿时必须同时恢复库存和 `SREM` 用户标记。

---

#### 4. 你这个补偿逻辑如何防止重复恢复库存？

答：

> 补偿前会先条件更新秒杀记录，只有状态为 `ORDER_CREATED` 的记录才能更新为 `CLOSED`。如果更新成功，说明这是第一次补偿，才继续恢复 Redis 库存、移除用户标记、回退 sold_count。如果更新影响行数为 0，说明已经补偿过或状态不匹配，就直接返回，避免重复补偿。

---

#### 5. Redis 查重后为什么还需要数据库记录？

答：

> Redis 主要用于高性能入口拦截，但它不是最终持久化事实来源，可能出现 key 过期、缓存清理或状态不一致。数据库中的秒杀记录用于持久化审计、订单关联和兜底校验。因此 Redis 负责快速判断，MySQL 负责最终记录和数据追溯。

---

#### 6. 为什么秒杀要引入 MQ？

答：

> 同步创建订单时，大量秒杀请求会同时打到数据库，容易造成数据库压力过大。引入 MQ 后，秒杀入口只负责 Redis Lua 原子预扣和投递消息，接口快速返回处理中。消费者按系统可承受的速度异步创建订单，从而削峰、解耦并提升系统抗压能力。

---

#### 7. 异步下单如何防止 MQ 重复消费？

答：

> 可以给秒杀记录增加 `PROCESSING` 状态。秒杀入口抢到资格后创建 `PROCESSING` 记录并发送 MQ。消费者消费消息时先检查记录状态，只有 `PROCESSING` 才允许创建订单，并在同一事务中更新为 `ORDER_CREATED`。如果消息重复投递，第二次消费发现状态已经变化，就直接跳过。

---

#### 8. RabbitMQ 和 RocketMQ 怎么选？

答：

> RabbitMQ 轻量、成熟、Spring Boot 集成简单，适合当前项目做秒杀异步下单、削峰和业务解耦。RocketMQ 吞吐更强，事务消息和交易场景能力更丰富，但架构和学习成本更高。当前项目优先选择 RabbitMQ，后续如果业务规模扩大、对事务消息和高吞吐要求更高，可以演进到 RocketMQ。
