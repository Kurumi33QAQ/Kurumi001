### 3. 今日关键问题与解答（一问一答）

#### Q1：为什么通知推送后还要推送未读数？

A：因为通知内容和未读数是两个不同维度。

通知内容告诉前端：

```text
发生了什么
```

未读数告诉前端：

```text
红点数量应该变成多少
```

例如用户收到订单通知：

```json
{
  "messageType": "MEMBER_NOTIFICATION",
  "title": "订单创建成功"
}
```

前端知道有一条新通知。

但如果要刷新红点，还需要：

```json
{
  "messageType": "UNREAD_COUNT_CHANGED",
  "unreadCount": 4
}
```

所以通知内容和未读数变化分开推，更清晰。

---

#### Q2：为什么标记已读后也要推送未读数？

A：因为标记已读会改变未读数量。

例如：

```text
原未读数 = 5
用户标记一条已读
新未读数 = 4
```

如果不推送最新未读数，前端红点可能不会及时更新。

所以：

```text
新增通知 -> 未读数增加
标记已读 -> 未读数减少
全部已读 -> 未读数变 0
```

这些场景都应该同步未读数。

---

#### Q3：当前通知中心为什么不是完整 IM？

A：因为当前系统是：

```text
业务事件通知用户
```

例如：

```text
订单创建成功
秒杀成功
系统通知
未读数变化
```

它不是用户之间自由对话。

完整 IM 还需要：

```text
会话表
聊天消息表
发送方/接收方
客服分配
消息顺序
已读回执
多端同步
撤回
重发
在线路由
离线消息同步
```

所以当前准确说法是：

```text
基于 WebSocket 的实时通知中心
```

不是：

```text
完整聊天系统
```

#### Q4：当前 WebSocket 通知中心有哪些局限？

A：主要有 5 个：

```text
1. 在线连接只存在本地内存，不支持多实例共享
2. token 通过 URL query 传递，生产环境最好改短期 ticket
3. 当前只支持单端连接，后连接会覆盖前连接
4. 推送失败只记日志，没有重试任务
5. 还没有管理员系统通知接口
```

这些不影响基础版闭环，但属于后续增强方向。

A：主要瓶颈是：

```text
数据库仍在请求主链路上
重复校验和扣库存不是 Redis 原子脚本

```

当前基础版已经能完成闭环，但高并发能力还不够。

---

#### Q7：为什么秒杀优化版要引入 MQ？

A：MQ 的作用是：

```text
削峰
异步
解耦
失败重试
```

没有 MQ 时：

```text
大量请求同时打数据库创建订单
```

有 MQ 后：

```text
秒杀入口只做 Redis 判断和投递消息
订单由消费者按系统能力异步创建
```

这样可以保护数据库，避免瞬间流量压垮订单系统。

---

#### Q8：为什么要用 Lua 脚本？

A：因为秒杀入口需要多个 Redis 操作：

```text
判断是否重复
判断库存是否充足
扣库存
记录用户已抢
```

如果分多次执行，中间可能存在并发窗口。

Lua 可以把这些操作放在 Redis 内部一次执行，具备原子性。

所以 Lua 适合做：

```text
防重复 + 查库存 + 扣库存 + 标记已抢
```

---

#### Q9：为什么优化版秒杀接口不应该直接返回订单 ID？

A：高并发优化版中，订单创建会异步执行。

所以接口无法马上返回订单 ID。

它应该返回：

```text
请求已受理，排队处理中
```

例如：

```json
{
  "status": "PROCESSING",
  "requestId": "xxx"
}
```

之后通过：

```text
WebSocket 推送秒杀结果
或查询秒杀结果接口
```

告诉用户最终成功或失败。

---

#### Q10：什么是最终一致性？

A：最终一致性是指系统多个组件的数据不一定瞬间一致，但通过补偿、重试、对账，最终达到一致。

秒杀优化版涉及：

```text
Redis 库存
MQ 消息
订单表
秒杀记录表
通知表
```

可能出现：

```text
Redis 扣库存成功但 MQ 发送失败
MQ 消费失败
订单创建成功但通知失败
Redis 库存和 MySQL sold_count 不一致
```

这些都需要补偿任务或对账任务修复。

### 5. 面试高频点（含标准答案）

#### 1. 你当前实现的是 WebSocket 通知中心还是 IM？

答：

> 当前实现的是 WebSocket 实时通知中心，不是完整 IM。它主要用于订单、秒杀、系统公告等业务事件通知，具备通知落库、历史查询、未读数统计、已读处理、在线实时推送和离线消息查询能力。完整 IM 还需要会话表、聊天消息表、客服分配、消息顺序、已读回执、多端同步等能力，后续可以在通知中心基础上扩展。

---

#### 2. 你的通知中心如何支持离线消息？

答：

> 我采用先落库再推送的设计。通知先保存到 `ums_member_notification` 表，如果用户在线，就通过 WebSocket 推送；如果用户离线，消息仍然保存在数据库里，用户后续登录后可以通过历史通知接口查询，因此离线消息不会丢。

---

#### 3. WebSocket 推送为什么要在事务提交后执行？

答：

> 因为订单创建、秒杀提交和通知创建通常在同一个事务里。如果通知刚 insert 就推送，而后续事务回滚，用户会收到数据库中不存在的通知，甚至订单也不存在。我使用 `TransactionSynchronizationManager` 注册 `afterCommit` 回调，只有事务真正提交成功后才推送，避免假消息。

---

#### 4. 当前 WebSocket 方案多实例下有什么问题？

答：

> 当前在线连接保存在本地 `ConcurrentHashMap` 中，只适合单体部署。多实例部署时，用户可能连接在实例 A，但业务请求打到实例 B，实例 B 找不到用户连接。后续需要用 Redis 维护在线状态，并用 Redis Pub/Sub 或 MQ 做跨实例消息路由。

---

#### 5. 秒杀基础版为什么还不能算高并发版本？

答：

> 当前基础版虽然用了 Redis 预扣库存，也有防重复、订单创建和补偿机制，但秒杀请求仍然同步创建订单，数据库仍在请求主链路上，而且 Redis 的重复校验和扣库存还没有通过 Lua 合并成原子操作，也缺少限流、削峰和最终一致性补偿。因此它是基础秒杀版本，不是完整高并发优化版。

---

#### 6. 秒杀为什么要引入 MQ？

答：

> MQ 用于削峰和异步下单。秒杀入口通过 Redis 判断库存和资格后，把下单请求投递到 MQ，接口立即返回处理中。消费者按数据库可承受的速度创建订单，这样可以避免大量请求同时打到 MySQL，提高系统抗峰值能力。

---

#### 7. 秒杀为什么要用 Lua？

答：

> Lua 可以把 Redis 中的多个操作合并为一个原子脚本。例如判断用户是否重复秒杀、判断库存是否充足、扣减库存、记录用户已抢，可以在 Redis 内部一次完成，避免多次 Redis 操作之间出现并发窗口，从而提升并发安全性。

---

#### 8. 优化版秒杀为什么返回“处理中”而不是订单 ID？

答：

> 优化版会采用 MQ 异步创建订单。秒杀接口只负责快速判断资格和投递消息，并不直接创建订单，所以不能马上返回订单 ID。接口应返回请求已受理或处理中，最终结果通过 WebSocket 推送或结果查询接口获取。

---

#### 9. 秒杀系统如何保证最终一致性？

答：

> 可以通过状态机、MQ 重试、死信队列、补偿任务和定时对账来保证最终一致性。例如 Redis 扣库存成功但 MQ 发送失败时，需要回滚 Redis 库存或通过本地消息表补偿；消费者创建订单失败时需要重试，失败后更新秒杀记录为失败并恢复库存；Redis 库存和 MySQL sold_count 不一致时通过定时对账修复。
