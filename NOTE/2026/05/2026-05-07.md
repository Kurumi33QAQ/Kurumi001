## 今日学习笔记（按时间顺序）2026/5/7

### 1. 今日总框架（先做了什么、为什么做）
今天主线是把“秒杀基础版”继续升级成“RabbitMQ 异步秒杀下单链路”，重点不是只把 MQ 加进去，而是让秒杀从同步下单演进到更接近真实项目的异步处理模型。

今天主要完成了六段内容：

1. RabbitMQ 环境问题排查与启动
- 先解决 RabbitMQ 连接失败问题，确认 RabbitMQ 服务正常启动。
- 目的：MQ 是后续秒杀异步下单的基础设施，后端必须能连上 RabbitMQ。

2. 秒杀入口从同步下单改为异步受理
- 秒杀接口不再直接创建订单。
- 改成 Redis 预扣成功后插入秒杀记录，然后发送 RabbitMQ 消息。
- 目的：秒杀入口变轻，快速返回，订单创建交给消费者异步处理。

3. RabbitMQ 消费者真正创建秒杀订单
- 消费者收到消息后调用业务 Service 创建订单。
- 目的：完成“入口投递消息 -> 消费者创建订单”的异步闭环。

4. 消费端幂等设计
- 新增 `PROCESSING`、`CREATING` 等状态。
- 消费者先通过条件更新抢占秒杀记录，防止重复消息创建重复订单。
- 目的：应对 RabbitMQ 至少一次投递导致的重复消费问题。

5. 消费失败补偿
- 订单创建前失败时，秒杀记录改为失败，Redis 秒杀库存回滚，用户参与标记移除，并发送失败通知。
- 目的：避免记录卡在处理中、库存无法恢复、用户无法再次参与。

6. RabbitMQ 可靠性增强
- 增加消费者有限重试配置。
- 增加死信交换机、死信队列。
- 增加开发环境队列数量查询接口。
- 目的：让失败消息可观察、可追踪，为后续补偿或人工排查做准备。

### 3. 今日关键问题与解答（一问一答）

#### Q1：为什么秒杀接口不再直接返回订单 ID，而是返回秒杀记录 ID？
A：因为改成 MQ 异步后，接口线程只负责“受理秒杀请求并发送消息”，订单还没有立即创建。此时只能返回 `recordId`，表示这次秒杀请求已经被系统接收。真正的订单 ID 会在消费者成功创建订单后写入秒杀记录。

#### Q2：为什么要加 `PROCESSING` 状态？
A：`PROCESSING` 表示 Redis 已经预扣成功，用户已经获得异步处理资格，但订单还没有创建。它是秒杀入口和 MQ 消费者之间的中间态。

#### Q3：为什么要加 `CREATING` 状态？
A：`CREATING` 用于消费端幂等。消费者收到消息后，先执行条件更新：
```sql
UPDATE sms_seckill_record
SET status = CREATING
WHERE id = ? AND status = PROCESSING
```
只有更新成功的消费者才能继续创建订单。重复消费时，状态已经不是 `PROCESSING`，更新失败，直接返回，从而避免重复创建订单。

#### Q4：RabbitMQ 为什么可能重复消费消息？
A：RabbitMQ 通常是“至少一次投递”模型。消费者处理成功但 ack 丢失、网络抖动、消费者宕机、手动重发等情况，都可能导致同一条消息再次投递。所以业务端必须做幂等。

#### Q5：为什么消费者不直接操作订单表，而是调用 `OmsOrderService.createSeckillOrder`？
A：订单创建属于 `oms` 模块职责。消费者属于消息接入层，秒杀 Service 负责秒杀业务编排，订单 Service 负责订单字段、状态和入库规则。这样模块边界清晰，避免跨模块乱写表。

#### Q6：`createOrderFromMessage` 为什么要先查数据库记录，再校验消息字段？
A：MQ 消息不能完全信任。数据库中的秒杀记录才是更可靠的业务事实。校验 `activityId`、`memberUsername`、`quantity` 可以防止消息和数据库记录不一致时创建错误订单。

#### Q7：为什么失败补偿只处理“订单创建前失败”？
A：如果订单已经创建成功，但后续更新秒杀记录失败，这种情况不能简单标记失败并回滚 Redis，否则会出现“订单存在但秒杀记录失败、库存又被加回”的数据不一致。所以订单创建前失败可以补偿；订单创建后失败应该抛异常，让事务回滚并交给 MQ 重试。

#### Q8：为什么要配置 RabbitMQ 有限重试？
A：如果消费者一直异常，消息无限回队列会造成日志刷屏、CPU 消耗和队列阻塞。有限重试可以让系统尝试自恢复，但不会无限消耗资源。

#### Q9：为什么要死信队列？
A：重试耗尽后，如果没有死信队列，失败消息可能丢失或不可观察。死信队列可以保存最终失败的消息，方便后续人工排查、后台补偿或重新投递。

#### Q10：为什么不马上写死信消费者自动处理失败消息？
A：死信消息一般代表系统已经自动处理失败了。贸然自动重投可能导致重复订单或反复失败。更稳的做法是先让死信消息可观察，再设计人工补偿或受控重投。

#### Q11：如果启动时报 `PRECONDITION_FAILED - inequivalent arg 'x-dead-letter-exchange'` 是什么原因？
A：RabbitMQ 中已经存在同名队列 `seckill.order.queue`，旧队列没有死信参数。RabbitMQ 不允许原地修改已有队列参数。解决方式是在 RabbitMQ 管理后台删除旧队列，然后重启后端让 Spring 重新声明队列。

#### Q12：为什么调试接口放在 `DevDebugController`？
A：因为它有 `@Profile("dev")`，只在开发环境加载。队列数量查询属于开发和排查工具，不应该作为正式业务接口暴露。

### 5. 面试高频点（含标准答案）

#### 1. 你为什么要在秒杀中引入 RabbitMQ？
答：秒杀请求瞬时并发高，如果接口线程直接创建订单，会给订单表、库存表和数据库连接池造成瞬时压力。引入 RabbitMQ 后，秒杀入口只做参数校验、Redis 原子预扣和消息投递，快速返回；订单创建交给消费者异步处理，从而实现削峰填谷。

#### 2. 你的秒杀接口现在返回的是什么？为什么不是订单 ID？
答：返回的是秒杀记录 ID，也就是 `recordId`。因为订单创建已经异步化，接口返回时订单可能还没有生成，所以不能返回订单 ID。消费者创建订单成功后，会把订单 ID 回写到秒杀记录里。

#### 3. 你如何防止 MQ 重复消费导致重复创建订单？
答：消费端使用秒杀记录状态机做幂等。入口插入 `PROCESSING` 状态记录，消费者收到消息后先执行条件更新，把状态从 `PROCESSING` 改为 `CREATING`。只有更新成功的消费者才能继续创建订单。重复消息再次消费时，状态已经不是 `PROCESSING`，更新行数为 0，直接返回。

#### 4. 为什么不直接依赖 RabbitMQ 保证消息只消费一次？
答：RabbitMQ 更接近“至少一次投递”模型，无法保证业务上的绝对只消费一次。网络抖动、ack 丢失、消费者异常等情况都可能导致重复投递。因此业务端必须实现幂等。

#### 5. `PROCESSING` 和 `CREATING` 的区别是什么？
答：`PROCESSING` 表示秒杀入口已经完成 Redis 预扣并发送 MQ，等待消费者处理；`CREATING` 表示某个消费者已经抢占了这条秒杀记录，正在创建订单。`CREATING` 是为了防止重复消费并发创建订单。

#### 6. 消费者创建订单为什么调用 `OmsOrderService`？
答：订单创建属于订单模块职责。秒杀模块只负责编排秒杀资格、记录和消息消费流程，具体订单字段、订单号、订单状态由 `OmsOrderService` 统一处理，避免跨模块直接操作订单表。

#### 7. 如果消费者创建订单失败怎么办？
答：如果订单尚未创建成功，就把秒杀记录标记为 `FAILED`，回滚 Redis 预扣库存和用户参与标记，并发送秒杀失败通知。如果订单已经创建后发生异常，则不直接补偿，而是抛异常让事务回滚，并由 MQ 重试机制处理。

#### 8. 为什么要配置有限重试？
答：有限重试可以处理临时性异常，例如短暂数据库抖动或网络波动。但如果无限重试，会导致消息不断回队列，造成日志刷屏、资源消耗和业务阻塞。所以设置最大重试次数，例如 3 次。

#### 9. 死信队列解决什么问题？
答：死信队列用于保存多次重试仍失败的消息，让失败消息不丢失、可观察、可排查。后续可以基于死信队列做人工补偿、后台重投或失败日志记录。

#### 10. 为什么不一开始就自动消费死信队列？
答：死信消息通常说明系统已经多次处理失败，自动消费可能导致反复失败或重复业务操作。更稳妥的方式是先观察死信消息，再通过人工确认或受控重投处理。

#### 11. RabbitMQ 队列参数变更为什么要删除旧队列？
答：RabbitMQ 不允许对已有同名队列原地修改关键参数，例如死信交换机参数。代码中声明了新的队列参数，但 RabbitMQ 中旧队列已经存在，就会报 `PRECONDITION_FAILED`。解决方法是删除旧队列，让应用重新声明。

#### 12. 当前秒杀 MQ 方案属于基础版还是高并发最终版？
答：当前属于“基础增强版”。它已经具备 Redis 原子预扣、MQ 异步下单、消费端幂等、失败补偿、有限重试和死信队列。但还不是最终高并发版，后续还可以加入限流、热点活动预热、消息确认机制优化、失败日志表、死信重投后台、分布式链路追踪和库存最终一致性校验。
