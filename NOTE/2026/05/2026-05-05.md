### 3. 今日关键问题与解答（一问一答）

#### Q1：RabbitMQ 这一步为什么先只做生产者，不直接做完整异步下单？

A：因为 MQ 链路复杂，最好拆开验证。

正确顺序是：

```text
1. Producer 能发送消息
2. Exchange 能路由消息
3. Queue 能收到消息
4. Consumer 能消费消息
5. Consumer 再接入创建订单业务
```

如果一口气写完，出错时很难判断问题在 MQ、反序列化、订单逻辑还是事务补偿。

---

#### Q2：Exchange、Queue、RoutingKey 分别是什么？

A：

```text
Exchange：交换机，接收生产者发来的消息，并根据规则路由
Queue：队列，真正存放等待消费的消息
RoutingKey：路由键，告诉 Exchange 消息应该去哪一个队列
```

本项目当前设计：

```text
Exchange = seckill.order.exchange
Queue = seckill.order.queue
RoutingKey = seckill.order.create
```

绑定关系：

```text
seckill.order.exchange 收到 routingKey 为 seckill.order.create 的消息
就投递到 seckill.order.queue
```

---

#### Q3：为什么选择 DirectExchange？

A：因为当前秒杀下单消息只需要精确路由。

也就是：

```text
seckill.order.create -> seckill.order.queue
```

DirectExchange 按 routing key 精确匹配，简单清晰，适合当前场景。

后续如果消息类型复杂，可以考虑 TopicExchange。

---

#### Q4：`RabbitTemplate` 是什么？

A：`RabbitTemplate` 是 Spring AMQP 提供的 RabbitMQ 操作工具。

类似于 Redis 中的：

```java
StringRedisTemplate
```

发送消息使用：

```java
rabbitTemplate.convertAndSend(exchange, routingKey, message);
```

它负责把消息发送到指定交换机，并携带路由键。

---

#### Q5：为什么要配置 `Jackson2JsonMessageConverter`？

A：因为我们发送的是 Java 对象：

```java
SeckillOrderMessage
```

RabbitMQ 实际传输的是字节数据。

`Jackson2JsonMessageConverter` 的作用是：

```text
发送时：Java 对象 -> JSON
消费时：JSON -> Java 对象
```

这样 Producer 和 Consumer 都可以直接操作 DTO，不用手动序列化和反序列化。

---

#### Q6：为什么 MQ 消息 DTO 不直接传完整 `SmsSeckillActivity`？

A：因为 MQ 消息应该尽量小而稳定。

如果直接传完整活动对象，会有问题：

```text
消息体过大
字段变化影响消费者
生产者和消费者耦合过深
不利于后续版本兼容
```

所以只传消费者创建订单需要的最小字段：

```text
recordId
activityId
productId
memberUsername
quantity
seckillPrice
createTime
```

#### Q9：为什么 Spring Boot 报 8080 被占用，但 `netstat` 查不到？

A：最终原因是 Windows 端口排除范围包含了 8080。

命令输出里有：

```text
8040 - 8139
```

而 8080 正好在其中。

这不是某个进程监听 8080，而是 Windows 系统保留了这段端口，普通程序不能绑定。

所以：

```text
netstat 查不到进程
但 Spring Boot 仍然无法启动 8080
```

解决方式是管理员 PowerShell 下删除排除范围：

```powershell
net stop winnat
netsh interface ipv4 delete excludedportrange protocol=tcp startport=8040 numberofports=100
netsh interface ipv6 delete excludedportrange protocol=tcp startport=8040 numberofports=100
net start winnat
```

### 5. 面试高频点（含标准答案）

#### 1. 你为什么在秒杀里引入 RabbitMQ？

答：

> 秒杀请求峰值很高，如果同步创建订单，大量请求会直接打到数据库，数据库压力会很大。我引入 RabbitMQ 是为了削峰和异步下单。秒杀入口先通过 Redis Lua 完成资格判断和库存预扣，成功后发送 MQ 消息，接口快速返回处理中。消费者再按系统承受能力异步创建订单，从而降低数据库瞬时压力。

---

#### 2. RabbitMQ 在你的秒杀链路中处于什么位置？

答：

> RabbitMQ 位于 Redis 资格判断之后、订单创建之前。秒杀入口通过 Lua 抢到资格后，会发送 `SeckillOrderMessage` 到 RabbitMQ。后续消费者监听队列，异步创建秒杀订单、更新秒杀记录，并通过 WebSocket 通知用户结果。

---

#### 3. 你为什么用 DirectExchange？

答：

> 当前秒杀下单消息路由关系比较简单，只需要把 `seckill.order.create` 这类消息精确投递到 `seckill.order.queue`，DirectExchange 按 routing key 精确匹配，简单清晰，适合当前场景。如果后续消息类型变多，可以再考虑 TopicExchange。

---

#### 4. RabbitMQ 中 Exchange、Queue、Binding 的关系是什么？

答：

> Producer 发送消息到 Exchange，Exchange 根据 routing key 和 Binding 规则把消息路由到 Queue，Consumer 从 Queue 中消费消息。Binding 就是 Exchange 和 Queue 之间的绑定规则。

---

#### 5. 为什么 MQ 消息要设计 DTO？

答：

> MQ 消息需要稳定、轻量，不能直接传完整实体对象。我设计 `SeckillOrderMessage` 只包含消费者创建订单所需字段，比如 recordId、activityId、productId、memberUsername、quantity 和 seckillPrice。这样可以降低消息体大小，也减少生产者和消费者之间的耦合。

---

#### 6. 你怎么验证 RabbitMQ 生产者发送成功？

答：

> 我先新增开发环境测试接口，调用后通过 RabbitMQ 管理页面查看 `seckill.order.queue` 的 Ready 数量是否增加，同时查看后端日志是否打印“秒杀下单消息已发送”。如果接口返回成功、日志正常、队列有消息，就说明 Producer、Exchange、RoutingKey 和 Queue 绑定关系是正确的。

---

#### 7. 消费者如何监听 RabbitMQ 队列？

答：

> 使用 Spring AMQP 的 `@RabbitListener` 注解监听指定队列。例如监听 `seckill.order.queue`，只要队列中有消息，Spring 就会自动调用消费者方法，并通过 JSON 消息转换器把消息反序列化为 `SeckillOrderMessage`。

---

#### 8. 为什么消费者先只打印日志？

答：

> 这是为了分步验证 MQ 链路。先确认消费者能正常接收并反序列化消息，再接入订单创建逻辑。这样如果后续出错，可以更清楚地区分是 MQ 消费问题，还是订单业务逻辑问题。
