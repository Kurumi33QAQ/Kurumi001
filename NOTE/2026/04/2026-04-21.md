**三、我今天问过的问题 + 我现在该怎么理解（重点是答案）**

1. 问：为什么不能复用 admin 登录，靠鉴权区分身份就行？  
   答：
- 技术上可做，但长期会很痛苦。
- admin 和 member 安全等级不同、业务字段不同、扩展方向不同。
- 分离账号域能减少耦合、降低风险、面试更像真实项目。  
  我现在的理解：短期省事不等于长期正确，电商项目最好分域。

2. 问：这一步代码到底改了什么，为什么要这么改？  
   答：
- 核心是 JWT 加 `userType`，并在刷新时保留这个字段。
- 这样后面过滤器才能知道“这是 admin token 还是 member token”。  
  我现在的理解：这是“身份隔离”的基础设施，不是花活。

3. 问：我用 Apifox 怎么验证才算真正跑通？  
   答：
- 不是只看登录 200，要验证完整链路：注册 -> 登录 -> me -> 刷新 -> 退出 -> me 401。
- 再加错误分支：密码错误锁定。  
  我现在的理解：接口测试要测“成功+失败+状态变化”，不是只测 happy path。

4. 问：为什么我要按小步推进，不一次性改完？  
   答：
- 一次性改太多，问题不好定位。
- 小步可以每步可编译、可验证、可讲解。  
  我现在的理解：小步是为了“学会+能讲”，不是为了慢。

**四、今天面试高频问题 + 直接可背答案**

1. 问：你如何实现买家登录安全？  
   答：  
   “我在买家登录实现了失败计数和锁定窗口策略。密码错误时累计 `loginFailCount`，达到阈值后写入 `lockExpireTime`，锁定期内直接拒绝登录；登录成功后清零失败次数并清空锁定时间，保证状态一致性。”

2. 问：为什么 JWT 要加 `userType`？  
   答：  
   “系统存在 admin 和 member 两类身份，仅靠用户名无法表达身份域。我在 JWT claim 加 `userType`，让过滤器可以按身份做鉴权分流，避免 token 交叉访问。”

3. 问：refresh token 时为什么要保留 `userType`？  
   答：  
   “刷新前后 token 的身份语义必须一致。若刷新后丢失 `userType`，会出现鉴权行为不一致，属于安全和一致性风险。”

4. 问：你怎么做退出登录？  
   答：  
   “我采用 Redis token 黑名单方案。logout 时把 token 按剩余有效期写入黑名单，过滤器每次请求先查黑名单，命中即拒绝认证，实现 token 立即失效。”

5. 问：`/me` 接口为什么不用前端传用户名？  
   答：  
   “前端参数可伪造，存在越权风险。`/me` 必须从 `SecurityContext` 读取当前认证主体，才能保证只返回当前登录用户的信息。”

6. 问：你今天这一版属于什么阶段？  
   答：  
   “属于买家认证基础版，已覆盖注册、登录、当前用户、刷新、退出、锁定。下一步会继续做更彻底的安全链路拆分和交易闭环。”
