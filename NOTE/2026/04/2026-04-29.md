### 3. 今日关键问题与解答（一问一答）

#### Q1：为什么通知中心要先做 HTTP 查询能力，再做 WebSocket？

A：因为 WebSocket 只是实时推送通道，不是消息系统本身。

如果没有通知表和查询接口，会有这些问题：

```text
用户离线时消息丢失
刷新后看不到历史通知
不能统计未读数
不能标记已读
不能补偿推送失败的消息
```

所以正确顺序是：

```text
通知落库
-> 历史通知查询
-> 未读数统计
-> 业务事件接入通知
-> WebSocket 实时推送
```

---

#### Q2：为什么订单创建成功后要调用通知 Service，而不是直接插通知表？

A：为了模块解耦。

订单模块只需要表达：

```text
订单创建成功，要发一条订单通知
```

至于通知如何保存、默认是否未读、后续是否 WebSocket 推送，都交给通知模块。

这样后续秒杀、系统公告、管理员推送也能复用同一个通知 Service。

---

#### Q3：订单通知为什么要放在订单插入成功之后？

A：因为通知需要绑定订单 ID。

订单没插入前，没有：

```text
order.getId()
```

只有订单真正落库后，通知的：

```text
businessId
```

才能设置为订单 ID。

这样前端点击通知时，后续可以跳转到订单详情页。

---

#### Q4：秒杀成功通知为什么不能放在 Redis 预扣成功后？

A：Redis 预扣成功只能说明用户暂时抢到了库存，并不代表完整秒杀成功。

完整秒杀成功至少要满足：

```text
秒杀记录插入成功
秒杀订单创建成功
秒杀记录更新为 ORDER_CREATED
活动 sold_count 更新成功
```

如果 Redis 预扣成功就发通知，后面订单创建失败，用户会收到错误的“秒杀成功”通知。

所以秒杀成功通知要放在业务状态稳定之后。

---

#### Q5：为什么这次只做秒杀成功通知，不做失败通知？

A：因为失败通知需要更细的策略。

例如：

```text
库存不足是否要通知？
重复秒杀是否要通知？
参数错误是否要通知？
活动未开始是否要通知？
```

如果所有失败都落库，会产生大量低价值通知。

所以基础版先做“成功类通知”，失败先通过接口响应返回。

后续如果引入 MQ 异步秒杀，再做：

```text
秒杀请求已受理
秒杀成功
秒杀失败
```

会更合理。

---

#### Q6：WebSocket 和 HTTP 最大区别是什么？

A：

HTTP 是：

```text
客户端主动请求
服务端被动响应
一次请求一次响应
```

WebSocket 是：

```text
客户端先建立长连接
连接保持不断
服务端可以主动向客户端推送消息
```

所以 WebSocket 适合：

```text
订单通知
秒杀结果通知
系统公告
未读消息红点
```

---

#### Q7：WebSocket 为什么也需要鉴权？

A：因为 WebSocket 连接一旦建立，服务端就可以通过这条连接持续推送消息。

如果不鉴权，任何人都可能连接：

```text
/ws/member
```

然后接收不属于自己的通知。

所以必须在连接建立前识别当前用户。

---

#### Q8：WebSocket 怎么识别当前用户？

A：在握手阶段解析 JWT。

当前基础版连接地址是：

```text
ws://localhost:8080/ws/member?token=买家JWT
```

后端流程是：

```text
HandshakeInterceptor 从 query 参数取 token
-> JwtTokenUtil 解析 token
-> 校验 userType 是否为 MEMBER
-> 校验 token 是否过期
-> 校验 token 是否在 Redis 黑名单
-> 解析出 memberUsername
-> 放入 WebSocketSession attributes
```

连接建立后，Handler 从：

```java
session.getAttributes()
```

中取出当前买家。

---

#### Q9：为什么 WebSocket token 放 query 参数，而不是 Authorization 请求头？

A：浏览器原生 WebSocket 不方便设置自定义请求头。

HTTP 请求可以带：

```text
Authorization: Bearer xxx
```

但浏览器 WebSocket 通常是：

```js
new WebSocket("ws://localhost:8080/ws/member?token=xxx")
```

所以基础版采用 query 参数传 token。

生产环境可以优化成：

```text
短期 ticket
一次性连接票据
WSS 加密传输
更严格的过期时间
```

---

#### Q10：为什么第一次连接失败，显示 1006？

A：第一次你在 Apifox query 参数里传的是：

```text
Bearer eyJ...
```

WebSocket URL 中空格会被编码成：

```text
Bearer%20eyJ...
```

后端可能无法正确识别 token，导致 JWT 解析失败，握手被拒绝。

Apifox 显示：

```text
1006 Abnormal Closure
```

它只是客户端看到的异常关闭。

真正原因是：

```text
握手阶段 token 校验失败
```

改成只传 token 本体后连接成功：

```text
token=eyJhbGciOiJIUzUxMiJ9...
```

---

#### Q11：为什么 `/ws/member` 要在 `SecurityConfig` 里放行？

A：因为 WebSocket 握手不是普通 Controller 接口。

我们当前设计是：

```text
SecurityConfig 放行 /ws/member
HandshakeInterceptor 自己做 JWT 鉴权
```

所以放行不是不鉴权，而是为了让请求进入 WebSocket 的握手流程。

真正的鉴权在：

```text
MemberWebSocketHandshakeInterceptor
```

里完成。

---

#### Q12：`MemberWebSocketSessionManager` 为什么用 `ConcurrentHashMap`？

A：因为 WebSocket 连接、断开、推送可能发生在不同线程。

普通 `HashMap` 线程不安全。

所以基础版使用：

```java
ConcurrentHashMap<String, WebSocketSession>
```

保存：

```text
memberUsername -> WebSocketSession
```

后续通知推送时，就可以根据用户名找到对应连接。

---

#### Q13：当前 WebSocket 在线连接管理支持集群吗？

A：不支持。

当前是单体基础版：

```text
内存 Map 保存在线用户连接
```

如果多实例部署，例如：

```text
8080 实例
8081 实例
8082 实例
```

用户连接可能在实例 A，但订单请求打到实例 B，实例 B 的内存 Map 找不到用户连接，就无法推送。

后续集群优化需要：

```text
Redis 记录在线状态
Redis Pub/Sub 或 MQ 做跨实例消息转发
服务实例只推送自己持有的 WebSocketSession
```

### 5. 面试高频点（含标准答案）

#### 1. 你这个通知中心是怎么设计的？

答：

> 我采用“先落库，再推送”的设计。通知表保存买家的订单通知、秒杀通知和系统通知，支持历史通知查询、未读数统计和标记已读。业务模块例如订单和秒杀不会直接操作通知表，而是调用通知 Service 创建通知。后续 WebSocket 只是实时推送通道，如果用户在线就推送，如果用户离线，通知仍然保存在数据库中，用户下次登录可以查询。

---

#### 2. 为什么通知要持久化，而不是只 WebSocket 推送？

答：

> WebSocket 只能解决在线实时推送，不能保证离线用户收到消息。如果只推送不落库，用户离线时消息会丢失，也无法支持历史通知、未读数统计和已读状态。因此我先把通知保存到数据库，再根据用户在线状态决定是否实时推送。

---

#### 3. 订单模块如何接入通知中心？

答：

> 订单创建成功后，订单 Service 调用通知 Service 创建一条订单通知，通知类型是 ORDER，businessId 绑定订单 ID。这样订单模块不需要关心通知表结构和后续 WebSocket 推送细节，只表达“订单创建成功，要发通知”这个业务事件，通知模块负责保存和推送。

---

#### 4. 秒杀成功通知为什么不能在 Redis 扣库存后立即发送？

答：

> Redis 预扣库存成功只能说明用户暂时占到了库存，不代表订单已经创建成功。秒杀完整成功还需要秒杀记录落库、订单创建成功、秒杀记录更新为已创建订单、活动已售数量更新成功。如果太早发送成功通知，后续订单创建失败会导致用户收到错误消息。所以我把秒杀成功通知放在订单和秒杀记录都处理成功之后。

---

#### 5. WebSocket 在你的项目里解决什么问题？

答：

> WebSocket 用来实现买家实时通知中心。比如订单创建成功、秒杀成功后，除了通知落库外，如果用户在线，服务端可以通过 WebSocket 主动推送给用户，避免前端不断轮询通知列表。它提升的是实时性和用户体验。

---

#### 6. WebSocket 怎么做 JWT 鉴权？

答：

> WebSocket 建立连接前会有一次 HTTP 握手。我在握手拦截器中从 query 参数读取 token，使用 JwtTokenUtil 解析 token，校验 token 是否过期、是否是 MEMBER 类型、是否在 Redis 黑名单中。校验通过后，把用户名写入 WebSocketSession 的 attributes。连接建立后，Handler 从 session attributes 中取出用户名，并把用户名和 WebSocketSession 绑定到在线连接管理器中。

---

#### 7. 为什么 WebSocket 不能直接用 Controller 里的 `Authentication`？

答：

> 普通 HTTP 请求会经过 Spring Security 过滤器，Controller 方法可以通过参数拿到 Authentication。WebSocket 是长连接，连接建立后不会每条消息都走普通 Controller 方法，所以不能直接依赖 Controller 的 Authentication 参数。我们需要在握手阶段完成身份识别，并把用户身份保存到 WebSocketSession attributes 中。

---

#### 8. 为什么 `/ws/member` 在 SecurityConfig 里 permitAll 了，还算安全？

答：

> `permitAll` 只是让 WebSocket 握手请求进入 Spring WebSocket 流程，不代表不鉴权。真正鉴权是在 `MemberWebSocketHandshakeInterceptor` 中完成的。它会校验 JWT、用户类型、过期时间和黑名单。因此鉴权位置从 Spring Security Filter 转移到了 WebSocket 握手拦截器。

---

#### 9. 当前 WebSocket 在线连接管理有什么局限？

答：

> 当前基础版使用本地 `ConcurrentHashMap` 保存 `memberUsername -> WebSocketSession`，适合单体应用。局限是不能直接支持多实例部署，因为不同实例之间无法共享内存连接状态。后续可以用 Redis 记录在线状态，用 Redis Pub/Sub 或 MQ 做跨实例消息路由。

---

#### 10. WebSocket 通知中心和完整 IM 系统有什么区别？

答：

> 当前实现的是实时通知中心，不是完整 IM。通知中心主要用于订单、秒杀、系统公告等业务事件，强调消息落库、未读数和在线推送。完整 IM 还需要会话表、聊天消息表、客服分配、已读回执、消息顺序、多端同步、在线路由等复杂能力。我的项目当前先实现通知中心，后续可以扩展为买家-客服实时会话。

