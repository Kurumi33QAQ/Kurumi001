### 3. 今日关键问题与解答（一问一答）

1. Q：为什么秒杀订单不能直接复用普通下单 `createOrder(...)`？  
   A：普通订单使用商品原价并扣普通库存；秒杀订单使用秒杀价，库存已经在 Redis 秒杀库存中预扣，不能再扣普通商品库存。

2. Q：为什么要加 `order_type/source_id`？  
   A：`order_type` 区分普通订单和秒杀订单；`source_id` 在秒杀订单里记录秒杀活动 ID。后续取消、统计、通知、售后都需要知道订单来源。

3. Q：为什么秒杀订单取消时不能恢复普通库存？  
   A：秒杀订单扣的是 Redis 秒杀库存和活动 `sold_count`，不是 `pms_product.stock`。恢复普通库存会导致库存数据错误。

4. Q：为什么需要 `SmsSeckillCompensationService`，不直接写在订单 Service 里？  
   A：订单模块只负责订单状态，秒杀库存、秒杀记录、活动已售属于 SMS 模块。把补偿逻辑放 SMS 服务里更符合模块边界。

5. Q：为什么用秒杀记录状态防重复补偿？  
   A：只有 `status=ORDER_CREATED` 的记录允许变成 `CLOSED`。第二次补偿时状态已经是 `CLOSED`，更新不到数据，从而避免重复回补库存。

6. Q：取消后再次秒杀，为什么还提示重复秒杀？  
   A：之前重复校验只按 `activity_id + member_username` 查到记录就拦截，没有区分 `status=3`。取消后记录仍存在，所以被拦截。

7. Q：`X-Idempotency-Key` 和秒杀重复判断是一回事吗？  
   A：不是。`X-Idempotency-Key` 当前用于普通订单防重复提交，保存在 Redis 中约 30 分钟。秒杀重复判断靠 `sms_seckill_record` 的活动 ID、用户名、状态。

8. Q：取消后允许重新秒杀会不会产生很多历史记录？  
   A：会。基础版保留历史记录用于审计，后续可通过限制重抢次数、活动结束归档、Redis 限流等方式控制数据膨胀。

### 5. 面试高频点（含标准答案）

1. 秒杀订单和普通订单如何区分？
- 通过 `order_type` 字段区分，`0` 普通订单，`1` 秒杀订单；秒杀订单的 `source_id` 存秒杀活动 ID。

2. 秒杀订单为什么不扣普通商品库存？
- 因为秒杀库存是活动维度库存，已经通过 Redis 预扣，普通商品库存不应该重复扣减。

3. 秒杀成功后为什么还要写 MySQL 订单？
- Redis 只适合高并发入口过滤，订单是交易事实，必须落到 MySQL 作为最终账本。

4. Redis 和 MySQL 混合使用时最大的注意点是什么？
- Redis 不参与 MySQL 事务，所以失败时要做手动补偿，例如订单创建失败要回滚 Redis 库存。

5. 如何防止秒杀订单重复补偿？
- 使用秒杀记录状态机，只有 `ORDER_CREATED` 状态能更新为 `CLOSED`，更新成功才执行库存回补，避免重复加库存。

6. 取消后是否允许重新秒杀怎么设计？
- 如果业务允许取消后重新抢，重复校验只拦截 `QUALIFIED` 和 `ORDER_CREATED`，不拦截 `CLOSED`。同时要调整唯一索引，否则数据库仍会阻止插入新记录。

7. 取消后重复参与会不会造成数据膨胀？
- 会。基础版可保留历史用于审计；后续用重抢次数限制、归档表、Redis 限流来优化。
