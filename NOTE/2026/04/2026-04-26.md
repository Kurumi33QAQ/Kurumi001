### 3. 今日关键问题与解答（一问一答）

1. Q：`return activityPage.convert(this::toPortalDTO);` 是什么意思？  
   A：它把分页里的每条 `SmsSeckillActivity` 实体转换成 `SmsSeckillActivityPortalDTO`，同时保留分页信息，如 `total/size/current/pages`。`this::toPortalDTO` 是方法引用，等价于 `activity -> this.toPortalDTO(activity)`。

2. Q：为什么不把 `activityStatus` 直接放到实体里？  
   A：实体映射数据库字段，`activityStatus` 是根据时间、库存、状态动态计算出来的展示字段，所以放到 DTO 更合理。

3. Q：为什么秒杀提交 DTO 只传 `activityId` 和 `quantity`？  
   A：秒杀商品、价格、库存必须以后端活动表为准，不能信任前端传来的价格或商品 ID。

4. Q：为什么要有秒杀记录表？  
   A：为了记录“谁抢过哪个活动”，防止重复秒杀。后续 MQ 异步下单时，它也能作为“秒杀资格记录”。

5. Q：为什么既先查重复，又加唯一索引？  
   A：先查能给用户友好提示；唯一索引能在并发下兜底，防止两个请求同时查不到后都插入成功。

6. Q：为什么 Redis 扣成功后，插记录失败要回滚 Redis 库存？  
   A：因为用户并没有真正获得资格，如果不回滚，库存会被白白扣掉，导致少卖。

7. Q：Redis 也是数据库，为什么 MySQL 扛不住的高并发 Redis 能扛？  
   A：Redis 是内存型 KV 数据库，适合简单高频原子计数操作。秒杀库存预扣本质是对一个 key 做原子递减，Redis 可以快速挡住大量无效请求；MySQL 负责最终落订单和记录，不承接所有抢购请求。

### 5. 面试高频点（含标准答案）

1. 为什么秒杀库存要单独存在秒杀活动表里？
- 秒杀价、秒杀库存、活动时间都属于活动维度，不能直接污染普通商品表。普通库存服务日常销售，秒杀库存服务活动限量抢购。

2. 为什么秒杀浏览接口可以放行，提交接口必须登录？
- 浏览活动是公开行为，抢购涉及用户资格、限购、防重复和订单创建，必须识别当前用户。

3. 为什么用 DTO 返回活动状态，而不是直接返回实体？
- DTO 用于接口展示，可以承载动态计算字段；实体应保持和数据库字段对应，避免领域边界混乱。

4. 秒杀活动状态如何判断？
- 先判断后台是否启用，再判断是否未开始、是否已结束、是否售罄，最后才是进行中。判断顺序体现业务优先级。

5. 如何防止同一用户重复秒杀？
- 基础版使用秒杀记录表记录 `activityId + memberUsername`，并加唯一索引兜底。业务层先查，数据库层防并发重复插入。

6. 为什么秒杀要用 Redis 预扣库存？
- Redis 的原子递减可以快速处理高频库存竞争，把大量失败请求挡在数据库前，避免 MySQL 同一行库存锁竞争和连接池耗尽。

7. Redis 预扣库存后如何保证不白扣？
- 如果后续插入秒杀记录失败，就调用回滚逻辑把 Redis 库存加回来。

8. 当前秒杀版本属于什么级别？
- 当前属于基础秒杀版：有活动校验、防重复、Redis 预扣库存，但还没有 Lua 脚本、MQ 异步下单、限流和最终一致性补偿。
