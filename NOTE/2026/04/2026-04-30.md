### 3. 今日关键问题与解答（一问一答）

#### Q1：WebSocket 为什么能推送给某个具体买家？

A：因为服务端保存了：

```text
memberUsername -> WebSocketSession
```

当买家连接：

```text
ws://localhost:8080/ws/member?token=买家JWT
```

后端会在握手阶段解析 JWT，拿到：

```text
memberUsername
```

然后在连接建立后保存：

```java
sessionManager.add(memberUsername, session);
```

所以后续通知推送时，只要知道买家用户名，就能找到他的 WebSocket 连接。

---

#### Q2：`MemberNotificationWebSocketSender` 是干什么的？

A：它是通知 WebSocket 发送器，专门负责“在线推送”。

它不负责创建通知，也不负责查通知列表。

它只做：

```text
拿到一条已经创建好的通知
判断用户是否在线
把通知转成 JSON
通过 WebSocketSession 发给用户
```

这样通知 Service 就不用直接操作 WebSocket 细节。

---

#### Q3：为什么通知 Service 不直接调用 `session.sendMessage`？

A：为了职责清晰。

通知 Service 的核心职责是：

```text
创建通知
查询通知
统计未读
标记已读
```

WebSocket 发送涉及：

```text
查在线连接
构造 JSON
TextMessage
sendMessage
异常处理
```

这些属于推送层细节。

所以拆成：

```text
UmsMemberNotificationServiceImpl：通知业务
MemberNotificationWebSocketSender：WebSocket 推送
```

更清晰，也更容易扩展。

---

#### Q4：为什么通知要先落库，再推送？

A：因为 WebSocket 只适合在线实时推送，不能保证离线用户收到消息。

如果只推送不落库：

```text
用户在线 -> 能收到
用户离线 -> 消息丢失
刷新页面 -> 历史消息没了
不能统计未读数
不能标记已读
```

所以当前设计是：

```text
先保存 ums_member_notification
再判断是否在线
在线就推送
离线就只保存数据库
```

这样既有实时性，也有可靠性。

---

#### Q5：为什么推送失败不能影响订单或秒杀？

A：因为 WebSocket 是体验增强，不是交易主流程。

主流程是：

```text
订单创建成功
秒杀成功
通知落库成功
```

如果用户网络断开导致 WebSocket 推送失败，但订单和通知都已经成功，就不能让订单回滚。

所以推送异常只记录日志：

```java
log.warn(...)
```

不继续往外抛。

这叫：

```text
弱依赖
```

也可以理解为：

```text
推送失败不影响交易链路
```

---

#### Q6：为什么不能通知一插入数据库就立刻推送？

A：因为订单和秒杀方法通常有事务。

例如订单创建：

```java
@Transactional(rollbackFor = Exception.class)
```

如果这样写：

```java
notificationMapper.insert(notification);
sendWebSocket(notification);
```

可能发生：

```text
通知已经推给用户
但后面业务报错
事务回滚
订单没了
通知也没了
用户却已经收到“订单创建成功”
```

这就是“假消息”。

所以要等事务真正提交成功后再推送。

---

#### Q7：`TransactionSynchronizationManager` 是什么？

A：它是 Spring 提供的事务同步工具。

可以理解成：

```text
Spring 事务生命周期的钩子管理器
```

它允许我们注册一些逻辑，在事务的特定阶段执行，例如：

```text
事务提交后 afterCommit
事务完成后 afterCompletion
```

今天用的是：

```java
TransactionSynchronizationManager.registerSynchronization(...)
```

注册一个：

```text
事务提交成功后再执行 WebSocket 推送
```

的回调。

---

#### Q8：`afterCommit()` 是什么作用？

A：`afterCommit()` 是事务提交成功后的回调方法。

当前代码含义是：

```java
@Override
public void afterCommit() {
    memberNotificationWebSocketSender.sendNotification(memberUsername, notification);
}
```

也就是说：

```text
只有数据库事务真正提交成功后，才推送 WebSocket 消息
```

如果事务回滚，`afterCommit()` 不会执行。

这能避免用户收到实际不存在的通知。

---

#### Q9：为什么要判断 `isSynchronizationActive()`？

A：因为 `createNotification` 不一定永远在事务里调用。

订单创建、秒杀提交通常有事务：

```text
isSynchronizationActive() = true
```

未来管理员直接发系统通知时，可能没有事务：

```text
isSynchronizationActive() = false
```

所以代码这样设计：

```text
如果有事务 -> afterCommit 后推送
如果没有事务 -> 直接推送
```

让通知 Service 更通用。

---

#### Q10：为什么推送 JSON 里要有 `messageType`？

A：因为 WebSocket 后续可能推多种类型的消息。

例如：

```text
MEMBER_NOTIFICATION：买家通知
UNREAD_COUNT_CHANGED：未读数变化
SECKILL_RESULT：秒杀结果
SYSTEM_NOTICE：系统公告
CUSTOMER_SERVICE_MESSAGE：客服消息
```

当前推送：

```json
{
  "messageType": "MEMBER_NOTIFICATION"
}
```

前端可以根据这个字段判断如何处理这条 WebSocket 消息。

---

#### Q11：`messageType` 和通知表里的 `type` 有什么区别？

A：两者不是一回事。

`messageType` 是 WebSocket 消息类型：

```text
这条 WebSocket 消息属于什么大类
```

例如：

```text
MEMBER_NOTIFICATION
```

`type` 是业务通知类型：

```text
这条通知来自哪个业务
```

例如：

```text
1：订单通知
2：秒杀通知
3：系统通知
```

可以理解为：

```text
messageType：通信层类型
type：业务层类型
```

---

#### Q12：为什么 `sendMessage` 外面要加 `synchronized(session)`？

A：因为同一个用户可能同时收到多条通知。

例如：

```text
订单通知
秒杀通知
系统通知
```

多个线程可能同时对同一个：

```java
WebSocketSession
```

调用：

```java
session.sendMessage(...)
```

这可能出现并发发送问题。

所以基础版对同一个 session 加锁：

```java
synchronized (session)
```

保证同一时间只有一个线程给这个连接发消息。

---

#### Q13：为什么发送前要判断两次 `session.isOpen()`？

A：第一次判断：

```java
if (session == null || !session.isOpen()) {
    return;
}
```

是为了快速过滤离线用户。

第二次判断：

```java
synchronized (session) {
    if (session.isOpen()) {
        session.sendMessage(...)
    }
}
```

是因为进入锁之前连接可能是打开的，但等待锁期间用户可能断开连接。

所以发送前再检查一次，避免给已关闭连接发送消息。

### 5. 面试高频点（含标准答案）

#### 1. 你的实时通知中心是怎么实现的？

答：

> 我实现的是一个基于“数据库持久化 + WebSocket 在线推送”的通知中心。订单创建、秒杀成功后会先调用通知 Service，把通知写入通知表。通知表支持历史查询、未读数统计和已读处理。通知落库后，如果用户在线，就通过 WebSocket 推送 JSON 消息；如果用户离线，通知仍然保存在数据库中，用户下次登录可以查询。

---

#### 2. 为什么不是直接 WebSocket 推送，而要先写数据库？

答：

> 因为 WebSocket 只能覆盖在线场景，不能保证离线用户收到消息。如果只推送不落库，用户离线时消息会丢失，也无法支持历史通知、未读数统计和已读状态。所以我采用先落库再推送的方案，兼顾可靠性和实时性。

---

#### 3. 如何判断用户是否在线？

答：

> 用户建立 WebSocket 连接时，服务端在握手阶段解析 JWT，拿到 memberUsername。连接建立后，把 `memberUsername -> WebSocketSession` 保存到 `ConcurrentHashMap` 中。推送通知时，根据 memberUsername 查询 session，如果 session 存在且处于打开状态，就认为用户在线并推送。

---

#### 4. WebSocket 推送失败会不会影响订单创建？

答：

> 不会。订单创建和通知落库是主流程，WebSocket 推送只是体验增强。推送失败时只记录 warn 日志，不向外抛异常。即使用户没有在线或连接断开，通知也已经保存在数据库中，用户后续仍然可以通过通知列表查询。

---

#### 5. 为什么要等事务提交后再推送 WebSocket？

答：

> 因为订单创建、秒杀提交和通知创建通常处于同一个事务中。如果通知 insert 后立刻推送，而后续事务回滚，用户会收到一条数据库中不存在的通知，甚至对应订单也不存在。为避免这种不一致，我使用 `TransactionSynchronizationManager.registerSynchronization` 注册 `afterCommit` 回调，只有事务真正提交成功后才推送 WebSocket 消息。

---

#### 6. `TransactionSynchronizationManager` 在这里起什么作用？

答：

> 它用于注册事务同步回调。我在通知创建成功后判断当前是否存在事务，如果存在，就注册一个 `afterCommit` 回调，让 WebSocket 推送在事务提交成功后执行；如果不存在事务，则直接推送。这样可以避免事务回滚时出现“假通知”。

---

#### 7. 为什么 WebSocket 消息里要有 `messageType`？

答：

> `messageType` 用于区分 WebSocket 消息大类。当前通知消息统一为 `MEMBER_NOTIFICATION`。后续如果增加未读数变化、秒杀结果、系统公告、客服消息等，都可以通过 `messageType` 让前端选择不同处理逻辑。

---

#### 8. 当前 WebSocket 推送方案有什么局限？

答：

> 当前方案是单体基础版，使用本地 `ConcurrentHashMap` 保存在线连接，适合单实例部署。如果后续多实例部署，用户连接可能在实例 A，而业务请求打到实例 B，实例 B 找不到用户 session。后续可以用 Redis 记录在线状态，并通过 Redis Pub/Sub 或 MQ 做跨实例消息路由。

---

#### 9. 你的通知中心和完整 IM 系统有什么区别？

答：

> 当前实现的是实时通知中心，不是完整 IM。通知中心主要推送订单、秒杀、系统公告等业务事件，关注消息持久化、未读数和在线推送。完整 IM 还需要会话表、聊天消息表、客服分配、消息顺序、已读回执、多端同步、离线消息路由等能力，复杂度更高。我的项目当前先实现通知中心，后续可以扩展为买家-客服会话。
