### 1. 今日总框架（先做了什么、为什么做）

今天主线是从“秒杀基础版收口”过渡到“WebSocket 通知中心阶段”。

分为四段：

1. 秒杀基础版补偿链路验证
- 验证秒杀订单取消后，秒杀记录会关闭，Redis 秒杀库存会恢复，活动已售数量会回退。
- 目的：确认秒杀不是只完成“抢购入口”，而是形成“秒杀成功 -> 创建订单 -> 订单关闭 -> 库存补偿”的完整闭环。

2. 阶段5 秒杀基础版收口
- 梳理当前秒杀能力属于“基础版”，不是高并发终极版。
- 目的：让你能在面试中准确表达项目能力边界，避免把基础版说成完整高并发秒杀架构。

3. Redis 连接失败问题排查
- 遇到 `Unable to connect to Redis`。
- 分析出问题出在 JWT 过滤器检查 token 黑名单时访问 Redis，但本地 Redis 没连上。
- 目的：理解 Redis 在当前项目里已经参与鉴权、登录、幂等、秒杀等多个核心链路。

4. 进入阶段6 WebSocket 通知中心
- 没有直接写 WebSocket，而是先设计通知中心的数据地基。
- 目的：先保证通知可持久化、可查询、可统计未读，再接实时推送。

### 2. 今日创建/修改的模块与文件（含作用与关系）

- 今日关注点：取消后重新秒杀时，重复校验只应拦截仍有效的秒杀记录，例如 `QUALIFIED`、`ORDER_CREATED`，不应拦截 `CLOSED`。

- 作用：秒杀订单关闭后的库存补偿。
- 核心职责：
    - 将秒杀记录状态改为 `CLOSED`
    - 恢复 Redis 秒杀库存
    - 回退活动 `sold_count`
- 今日关注点：补偿要防止重复执行，否则可能导致库存被多次恢复。

3. `OmsOrderServiceImpl`
- 作用：订单创建、取消、超时关闭。
- 今日关注点：普通订单关闭和秒杀订单关闭走不同补偿逻辑。
- 普通订单关闭：恢复商品库存。
- 秒杀订单关闭：恢复秒杀库存，并关闭秒杀记录。

4. `SmsSeckillRecordStatus`
- 作用：统一维护秒杀记录状态常量。
- 当前状态：
    - `QUALIFIED = 0`：已获得秒杀资格，尚未创建订单
    - `ORDER_CREATED = 1`：已创建订单
    - `FAILED = 2`：秒杀失败
    - `CLOSED = 3`：订单已取消或关闭，库存已补偿

### 3. 今日关键问题与解答（一问一答）

#### Q1：秒杀基础版现在到底完成了什么？

A：现在已经完成了秒杀基础版闭环：

```text
查询秒杀活动
-> 校验活动时间
-> 校验是否重复秒杀
-> Redis 预扣秒杀库存
-> 创建秒杀记录
-> 创建秒杀订单
-> 订单取消或超时关闭
-> 关闭秒杀记录
-> 恢复 Redis 秒杀库存
-> 回退活动 sold_count
```

这说明现在不是只有一个“秒杀接口”，而是有交易闭环。

---

#### Q2：为什么说当前秒杀系统只是基础版，不是高并发优化版？

A：因为当前版本虽然使用 Redis 预扣库存，已经避免所有请求直接打 MySQL，但还没有引入更完整的高并发架构。

当前基础版有：

```text
Redis 预扣库存
数据库记录防重复
同步创建订单
订单关闭补偿
```

还没有：

```text
MQ 异步下单
Lua 脚本保证 Redis 内部多步骤原子性
接口限流
热点活动本地缓存
削峰队列
最终一致性补偿任务
分布式 WebSocket 路由
```

所以面试时要准确说成“秒杀基础版”。

---

#### Q3：Redis 报 `Unable to connect to Redis` 是什么意思？

A：意思是 Spring Boot 后端尝试连接 Redis，但连接失败。

关键错误是：

```text
Connection refused: localhost/127.0.0.1:6379
```

表示程序去连本机 `6379` 端口，但这个端口没有 Redis 服务在监听。

常见原因：

```text
Redis 没启动
Docker 没启动
Redis 容器停止了
Redis 端口不是 6379
application.yml 配置和实际 Redis 地址不一致
```

---

#### Q4：为什么我明明不是在测秒杀，也会访问 Redis？

A：因为请求进入系统后会先经过 JWT 鉴权过滤器。

链路是：

```text
请求进入
-> JwtAuthenticationTokenFilter
-> 解析 Authorization token
-> TokenBlacklistService.contains(token)
-> RedisTemplate.hasKey(...)
-> 访问 Redis
```

你之前做过退出登录 token 黑名单机制，所以只要请求带 token，就可能检查 Redis 黑名单。

因此，Redis 不只是秒杀用，也参与鉴权链路。

---

#### Q5：为什么 WebSocket 通知中心不直接先写 WebSocket？

A：因为 WebSocket 只是实时推送通道，不是消息系统本身。

如果只推送不落库，会有问题：

```text
用户离线时消息丢失
刷新页面后看不到历史消息
不能统计未读数
不能标记已读
不能做通知列表
```

所以正确顺序是：

```text
先设计通知表
再做通知查询和未读数
最后接 WebSocket 实时推送
```

---

#### Q6：为什么通知表放在 `ums` 模块？

A：因为通知是发给买家用户的，属于用户侧能力。

模块职责上：

```text
pms：商品
oms：订单
sms：营销/秒杀
ums：用户、买家、权限、用户消息
```

通知虽然可能由订单或秒杀触发，但通知的归属对象是买家，所以放在 `ums` 更合理。

---

#### Q7：为什么通知表里既有 `member_id`，又有 `member_username`？

A：`member_id` 是买家主键，适合做稳定关联。

`member_username` 是为了适配当前项目的鉴权上下文，因为现在买家接口里常用：

```java
authentication.getName()
```

拿到当前登录用户名。

所以查询我的通知时可以直接按：

```sql
member_username = 当前登录用户名
```

这样和现有买家订单查询风格一致，改动更小。

---

#### Q8：为什么通知表要有 `business_id`？

A：用于把通知和具体业务关联起来。

例如：

订单创建通知：

```text
type = 1
business_id = orderId
```

秒杀成功通知：

```text
type = 2
business_id = seckillActivityId 或 orderId
```

这样前端点击通知时，可以跳转到对应订单详情或秒杀结果页。

---

#### Q9：为什么需要 `read_status`？

A：用于支持未读消息能力。

例如统计未读数：

```sql
SELECT COUNT(*)
FROM ums_member_notification
WHERE member_username = 当前用户
  AND read_status = 0
  AND delete_status = 0;
```

没有 `read_status`，就无法区分已读和未读，也无法实现通知红点。

---

#### Q10：为什么通知类型和已读状态用常量类，不直接用数字？

A：直接写数字会变成魔法值。

例如：

```java
notification.setType(1);
notification.setReadStatus(0);
```

别人看不出来 `1` 和 `0` 是什么意思。

用常量类后：

```java
notification.setType(UmsMemberNotificationType.ORDER);
notification.setReadStatus(UmsMemberNotificationReadStatus.UNREAD);
```

可读性更强，也和当前项目已有的 `OmsOrderStatus`、`SmsSeckillRecordStatus` 风格一致。

### 5. 面试高频点（含标准答案）

#### 1. 你这个秒杀系统为什么能防止基础超卖？

答：

> 我没有让所有秒杀请求直接扣 MySQL 库存，而是先使用 Redis 做秒杀库存预扣。Redis 的递减操作是单线程模型下的原子操作，可以快速判断库存是否充足。只有 Redis 预扣成功后，才会继续创建秒杀记录和订单。订单取消或超时关闭时，再通过补偿逻辑恢复 Redis 秒杀库存，并回退活动已售数量。因此在基础并发场景下，可以避免大量请求直接竞争 MySQL 库存，降低超卖风险。

---

#### 2. 当前秒杀系统为什么还是基础版？

答：

> 当前版本使用 Redis 预扣库存、数据库记录防重复、同步创建订单和关闭补偿，已经具备基础秒杀闭环。但它还没有引入 MQ 异步下单、Lua 脚本合并 Redis 判断和扣减、接口限流、削峰队列、热点缓存和最终一致性补偿任务。因此我会把它定义为秒杀基础版，适合中小并发场景。后续优化方向是引入 MQ 削峰异步创建订单，并用 Lua 脚本保证库存扣减和重复校验的 Redis 原子性。

---

#### 3. 秒杀订单取消后为什么要补偿库存？

答：

> 秒杀成功后会占用秒杀库存并创建待支付订单。如果用户取消订单或者订单超时未支付，就说明这次交易没有完成。为了避免秒杀库存被无效订单长期占用，需要在订单关闭时恢复秒杀库存，同时把秒杀记录状态改为已关闭。这样其他用户或者该用户在规则允许的情况下可以继续参与秒杀。

---

#### 4. 如何避免秒杀订单被重复补偿？

答：

> 补偿时不能只根据订单 ID 直接恢复库存，而是要带状态条件更新秒杀记录。比如只有当前记录状态是 `ORDER_CREATED` 时，才允许改成 `CLOSED` 并恢复库存。如果记录已经是 `CLOSED`，说明之前已经补偿过，再次触发补偿时不会重复恢复库存。这个状态条件相当于一个幂等保护。

---

#### 5. Redis 在你项目里有哪些作用？

答：

> Redis 在项目里承担了多个职责。第一，JWT 退出登录后会把 token 加入 Redis 黑名单，请求进入时过滤器检查 token 是否失效。第二，买家登录失败锁定可以用 Redis 或数据库辅助控制。第三，普通订单创建使用 Redis 做幂等锁，防止重复提交。第四，秒杀使用 Redis 做库存预扣，降低 MySQL 压力并减少超卖风险。后续 WebSocket 也可以用 Redis 维护在线状态或多实例消息路由。

---

#### 6. 为什么 Redis 宕机会导致接口报错？

答：

> 因为请求进入系统后会经过 JWT 过滤器，过滤器会检查 token 是否在 Redis 黑名单中。如果 Redis 不可用，黑名单检查就会失败，从而导致接口异常。这个问题说明 Redis 已经处于核心链路中，生产环境需要考虑 Redis 高可用、异常降级、超时配置和监控告警。

---

#### 7. WebSocket 和 HTTP 有什么区别？

答：

> HTTP 是客户端主动请求、服务端被动响应，一次请求对应一次响应。如果前端想知道订单状态变化，只能轮询。WebSocket 是客户端和服务端建立长连接，连接建立后服务端可以主动向客户端推送消息。因此 WebSocket 适合订单通知、秒杀结果通知、系统公告这类实时性要求较高的场景。

---

#### 8. 为什么通知中心要先落库，再 WebSocket 推送？

答：

> WebSocket 只解决在线实时推送问题，不能保证离线用户收到消息。如果只推送不落库，用户不在线时消息会丢失，也无法支持历史通知和未读数统计。因此通知中心应该先把消息保存到数据库，再判断用户是否在线，在线就推送，不在线就等用户后续查询历史通知。

---

#### 9. 通知表为什么需要 `read_status` 和 `delete_status`？

答：

> `read_status` 用于区分已读和未读，支持未读数统计和消息红点。`delete_status` 用于逻辑删除，用户删除通知时不直接物理删除，方便后续审计、排查问题或做数据归档。这是后台系统中比较常见的设计方式。

---

#### 10. 你现在做的是通知中心还是 IM 系统？

答：

> 当前做的是实时通知中心，不是完整 IM 系统。通知中心主要服务于商城业务事件，比如订单创建、订单关闭、秒杀成功、秒杀失败和系统公告。它强调业务消息的持久化、未读数和在线推送。完整 IM 系统还需要会话表、消息表、客服分配、在线路由、已读回执、消息顺序、多端同步等能力，复杂度更高，后续可以在通知中心稳定后扩展。
