### 2. 阶段一：认证与安全（先做“能安全登录”）
创建/完善内容：
- JWT 登录、`/demo/token/refresh`、`/demo/logout`
- Redis 黑名单：退出后 token 即时失效
- 登录失败锁定：5次失败锁10分钟
- 成功登录后重置失败次数并清锁定时间

关键实现点：
- 黑名单 key 带 TTL，和 token 剩余有效期对齐
- 过滤器鉴权前先查黑名单
- 锁定状态持久化字段：
    - `login_fail_count`
    - `lock_expire_time`

你问过的重点与答案：
- “多个 token 并存是否正常？”  
  正常（多会话策略），但安全上可再做单会话/版本号策略。
- “为什么清空锁定时间没生效？”  
  `updateById` 默认可能不更新 null，改 `LambdaUpdateWrapper.set(..., null)` 解决。

面试可讲：
- “我补齐了无状态认证的主动失效能力与防爆破策略。”

### 3. 阶段二：RBAC 闭环（从“能鉴权”到“可管理”）
完成能力：
- 用户分配角色（覆盖式）
- 角色分配资源（覆盖式）
- 用户角色回显、角色资源回显
- 权限汇总接口（用户+角色名+权限点）
- 角色/资源 options 下拉接口

缓存一致性：
- 权限缓存从本地 Map 迁到 Redis
- 角色/资源变更后自动清相关用户缓存
- 验证：不重新登录即可权限即时变化

你问过的重点与答案：
- “为什么带 token 不一定只能查自己？”  
  token 解决认证，不自动做数据权限隔离；隔离要业务层显式约束。
- “为什么权限改了接口还返回旧值？”  
  命中缓存；清缓存后读取新权限。

面试可讲：
- “我做了 RBAC 的读写闭环和缓存一致性，而不是只做静态鉴权。”

### 6. 阶段五：业务接口权限化（非常关键）
做了什么：
- 给 `pms/oms` 核心接口加 `@PreAuthorize`
- 新增权限点（`pms:product:read/write`、`oms:order:read/write`）
- 绑定管理员角色并验证：
    - 有权限 -> 200
    - 无权限 -> 403

联调坑点（你刚遇到）：
- 上传接口 404 的真实原因是路径前缀叠加  
  `@RequestMapping("/demo/product")` 会给该 Controller 下所有方法加前缀。  
  如果前端配 `/demo/file/upload`，而后端实际是 `/demo/product/...`，就会 404。

面试可讲：
- “我不仅写了接口，还把业务接口纳入 RBAC 做了真实权限验证。”

### 8. 今日高频问题速记（复习版）
1. `No static resource demo/file/upload` 是什么？  
   后端没匹配到该路由，通常是 Controller 前缀叠加导致真实路径不一致。

2. 为什么刚登录 token 也可能 401？  
   可能 token 过期/被拉黑/Redis未启动/请求头不规范。

3. 为什么权限变更后同 token 也能生效？  
   token 只存身份，权限是动态加载；缓存失效后读取新权限。
