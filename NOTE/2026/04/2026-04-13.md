### 2. 关键文件/模块变化（含作用与关系）

#### A. `JwtTokenUtil`（`security/util`）
- 作用：生成 token、解析 token、判断是否过期、校验 token。
- 今天重点：
    - 从旧写法升级到 `jjwt 0.11.5` 标准写法（`parserBuilder`）
    - 用 `SecretKey` + `Keys.hmacShaKeyFor(...)` 处理签名 key
- 关系：被登录接口和 JWT 过滤器共同依赖。

#### B. `pom.xml`（JWT依赖）
- 作用：提供 JWT 相关类。
- 今天重点：
    - 使用 `jjwt-api + jjwt-impl + jjwt-jackson` 组合
- 学到：IDE 红线可能和运行时 classpath 不一致，需 Maven Reload/重建索引。

#### C. 登录接口返回结构
- 相关 DTO：`LoginResponseDTO`（token/tokenHead/adminInfo）
- 作用：登录一次返回前端初始化所需完整信息。
- 关系：`DemoController` 调 service 校验后，调用 `JwtTokenUtil` 签发 token，再组装响应 DTO。

#### D. `JwtAuthenticationTokenFilter`（`security/component`）
- 作用：从请求头读取 JWT，校验通过后把认证信息放进 `SecurityContextHolder`。
- 关系：由 `SecurityConfig` 注入过滤链，在受保护接口前执行。

#### E. `SecurityConfig`（`security/config`）
- 作用：定义访问规则与过滤链。
- 今天重点：
    - 无状态会话（STATELESS）
    - 放行登录/注册等接口，其余要求认证
    - 接入 JWT 过滤器
    - 接入 401/403 自定义处理器

#### F. `RestAuthenticationEntryPoint` 与 `RestfulAccessDeniedHandler`
- 作用：统一返回 401/403 的 JSON，避免默认空白/HTML响应。
- 关系：由 `SecurityConfig` 的 `exceptionHandling(...)` 指定生效。

#### G. 方法级权限控制
- `@EnableMethodSecurity` + `@PreAuthorize("hasAuthority('admin:read')")`
- 作用：在方法层做精细权限控制。
- 今天先用轻量方案：
    - `demo_user` 返回 `admin:read`
    - 其他用户返回空权限

### 3. 今日你提出的关键问题与解答（Q&A）

#### Q1：为什么项目能运行、接口也通，但 `JwtTokenUtil` 还是爆红？
A：IDE 索引模型可能未刷新，运行用的是已编译 classpath，编辑器红线用的是本地索引，可能短暂不一致。Maven Reload + Rebuild + 重启IDE可恢复。

#### Q2：这版 JWT 工具类和之前有什么区别？
A：核心升级是新版 JJWT API（`parserBuilder`）、`SecretKey` 规范签名、JDK21兼容。不是仅“换语法”，而是消除旧库兼容隐患。

#### Q3：`Authentication` 是什么？
A：Spring Security 的当前认证对象，表示“当前请求是谁、有什么权限”。Controller 里可直接注入读取当前用户身份。

#### Q4：为什么改 `/me` 接口前后都可能是 403/200，看起来没差？
A：表面状态码可能相同，但身份来源已从“前端参数 username”变成“SecurityContext 中认证身份”，安全语义更正确，防伪造更强。

#### Q5：为什么要做 401/403 JSON 统一？
A：前后端协作需要稳定协议，安全异常也应走统一返回结构，便于前端处理与日志排障。

#### Q6：`authorityList.stream().map(...GrantedAuthority...)` 这段在干嘛？
A：把字符串权限（如 `admin:read`）转换为 Spring Security 可识别的 `GrantedAuthority` 集合，供 `@PreAuthorize` 判定权限。

### 5. 面试高频点（含可讲答案）

1. JWT 过滤器放在哪，为什么？
- 放在 `UsernamePasswordAuthenticationFilter` 前，用于先完成 token 认证并建立安全上下文。

2. 401 和 403 的区别？
- 401：未认证（没登录/无效token）
- 403：已认证但权限不足

3. 为什么要从 `SecurityContext` 取当前用户，而不是前端传 username？
- 防伪造身份，后端以认证结果为准，安全边界更清晰。

4. 为什么要统一安全异常返回 JSON？
- 与业务接口协议一致，前端处理更稳定，排障更高效。

5. 方法级权限怎么做？
- 开启 `@EnableMethodSecurity`，用 `@PreAuthorize` 按 authority 控制接口访问。

6. 这套权限为什么是“可扩展”的？
- 当前权限来源先轻量（按用户名），后续可替换为数据库 RBAC 查询，不改鉴权框架主干。
