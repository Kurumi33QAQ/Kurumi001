### 2. 本次创建/改造的核心模块与作用

#### A. Redis 化 Token 黑名单（会话治理）
- `TokenBlacklistService` 从 `ConcurrentHashMap` 改为 `StringRedisTemplate`。
- key 设计：`security:blacklist:token:<token>`。
- 写入时按 token 剩余生命周期设置 TTL。

作用：  
解决“重启丢失黑名单、单机有效、多实例不一致”的问题。

---

#### B. Redis 化权限缓存（鉴权性能 + 一致性）
- `UmsAdminServiceImpl#getAuthorityList` 改为：
    - 先查 Redis
    - 未命中查 DB
    - 回写 Redis（TTL 30 分钟）
- `evictAuthorityCache(username)` 改为删 Redis key。
- `getAuthorityCacheSize()` 改为统计 Redis 前缀 key。
- 清理掉旧 `authorityCache`（本地 Map）痕迹。

作用：  
权限缓存跨实例共享，且缓存失效机制明确。

---

#### C. 用户-角色管理闭环（RBAC 写能力）
新增：
- `AdminRoleAssignDTO`

### 3. 关键理解点（Q&A）

Q1：为什么 token 黑名单迁到 Redis 后更合理？  
A：Redis 支持 TTL、多实例共享、重启不丢；内存版只适合单机学习。

Q2：为什么第二步就 401？  
A：可能 token 已在黑名单，或已过期；Redis 黑名单不会因应用重启失效。

Q3：为什么不重新登录，权限也会变化？  
A：你 token 只存身份（username），权限是请求时动态加载；缓存清理后会读到新权限。

Q4：为什么权限改了但先看到旧权限？  
A：命中了 Redis 缓存；清缓存后才会读数据库新值并回写缓存。

Q5：`@Transactional` 为什么要加在分配逻辑上？  
A：覆盖式分配是“删旧+插新”组合操作，必须保证原子性，避免中间态脏数据。
